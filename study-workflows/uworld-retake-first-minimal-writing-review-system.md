# UWorld Retake-First, Minimal-Writing Review System

## Status

- **Status:** Canonical workflow
- **Created:** 2026-06-09
- **Use case:** Speed up UWorld review while preserving the reasoning repairs that actually improve future performance.

## Core principle

Retaking replaces excessive transcription, not diagnosis.

The goal of review is not to produce a long document for every question. The goal is to identify the failure, preserve the smallest reusable correction, and then prove the correction can be applied under time pressure.

## Review sequence

### 1. Rapid explanation check

For each question:

1. Read the explanation quickly.
2. State why the correct answer is correct.
3. Identify the exact failure category when relevant:
   - content gap
   - passage or data interpretation error
   - wording misread
   - calculation or setup error
   - changed away from the correct answer
   - correct answer reached through fragile reasoning
4. Write only a one- or two-line repair rule when the lesson is reusable.

### 2. Immediate mini-retake

Redo the question without looking at the explanation.

- Explain why the correct choice works.
- Explain why the tempting wrong choice fails.
- Do not count simple answer recognition as mastery.

### 3. Timed full-test retake

Use a timed retake to train:

- urgency
- passage navigation
- decision speed
- endurance
- willingness to move on

Flag any question answered correctly only because the answer was remembered.

### 4. Delayed retake

Retake after several days, preferably with shuffled questions when available.

Only questions that are missed, uncertain, or recognized from memory require another detailed review.

## Minimal question record

Preserve only:

- **What I thought**
- **Where the reasoning broke**
- **Corrected reasoning**
- **One reusable takeaway**

Everything else may remain verbal unless it reveals a recurring weakness.

## Completion rule

A question is not mastered merely because the explanation feels familiar.

A question is considered repaired when the student can:

1. answer it without the explanation,
2. state the tested principle,
3. reject the tempting distractor,
4. apply the same principle to a changed scenario,
5. do so within realistic timing.

## Status labels for review files

Use these labels consistently:

- **Pending review:** Test taken, review not started.
- **In progress:** Review started but unresolved questions remain.
- **Completed review guide:** All reviewed questions have a documented outcome or repair rule.
- **Retake pending:** Review completed; timed or delayed retake still needed.
- **Retake verified:** Delayed application confirms the repair.
- **Archived:** Older version preserved but replaced by a newer canonical file.

## GitHub storage boundary

Store in GitHub:

- reusable workflows
- concise correction guides
- sticky-note rules
- status indexes
- versioned Markdown summaries

Keep outside GitHub unless deliberately excerpted:

- raw UWorld passages
- complete proprietary question text
- full answer explanations
- screenshots
- large generated DOCX files

## Non-negotiable rule

**Extract the minimum useful lesson, then repeatedly prove that it can be applied under time pressure.**
